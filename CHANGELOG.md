## [2.0.0] — 2026-04 — V2 SHIPS 🚀

V2 closes out the project: deeper modeling, ensembles, drift monitoring, a
clinician dashboard, and end-to-end documentation.

### Added

- **Phase 13 — V2 NN & AutoML benchmark.** PyTorch MLP, TabNet, FT-Transformer,
  and FLAML AutoML benchmarked under the V1 protocol. Results in
  `reports/v2_leaderboard.csv` and figure 29.
- **Phase 14 — Stacking + soft-Voting ensembles** of the top-3 boosted trees.
  Results in `reports/ensembles_summary.csv` and figure 30.
- **Phase 15 — Trust extras.** LIME local explanations (3 high-risk + 3 low-risk
  encounters) under `reports/lime/`; Evidently train-vs-holdout drift report at
  `reports/drift_report.html` + machine-readable summary in
  `reports/drift_summary.json`.
- **Phase 16 — Flask dashboard.** Modern medical-SaaS app shell (sidebar +
  topbar + light/dark) reusing the production `Predictor`. Score / Leaderboards
  / Fairness / Drift / Method pages, JSON `/api/predict` endpoint,
  Dockerfile.flask + Procfile.
- **Phase 17 — Polish & release.**
  - `reports/report.html` — self-contained blog-style project report
    (~9 MB, all 25+ figures embedded as base64), generated by
    `scripts/build_report.py`.
  - `docs/architecture.md` — Mermaid system diagram + sequence diagram +
    module map.
  - `reports/dashboard.png` + `reports/demo.gif` — hero screenshot and
    animated walkthrough, generated headlessly by `scripts/record_demo.py`
    (Playwright + Pillow).
  - README polished with new badges, hero image, demo GIF, artifacts table.

### Honest negative results

- Deep tabular models (MLP / TabNet / FT-Transformer) and FLAML AutoML did
  **not** beat the tuned XGBoost on this cohort.
- Stacking and soft-voting ensembles added ~0.0001 PR-AUC over the best base
  learner — within noise. The deployed model remains tuned XGBoost.

### Changed

- Top-level README: status flipped from V1 → V2 shipped, added artifacts table.

## [1.0.0] — 2026-04 — V1 SHIPS 🚢

First public release. End-to-end production-grade pipeline: a tuned, calibrated,
fairness-audited 30-day inpatient readmission risk model, deployed as a
containerized FastAPI service with a CI pipeline.

### Added

- **Data & cohort** (Phase 1–3) — 1.33M-encounter inpatient cohort built from
  CMS DE-SynPUF (20 sample files), 9.61% positive rate, patient-grouped 70/15/15
  splits with no patient leakage.
- **Feature pipeline** (Phase 4) — 29-feature `ColumnTransformer` (numeric
  scaling, binary cast, low-card OneHot, high-card TargetEncoder) producing 105
  model-facing features.
- **Baseline benchmark** (Phase 5) — 7 models (LR, RF, ExtraTrees, HistGB,
  XGBoost, LightGBM, CatBoost) on the full 928k-row train, MLflow-tracked, with
  the 28-plot visual gallery in `reports/figures/`.
- **Hyperparameter tuning** (Phase 6) — Optuna TPE with median pruner, 20
  trials × 3-fold StratifiedGroupKFold on a 300k subsample, top-3 GBMs tuned.
- **Calibration + cost-based threshold** (Phase 7) — uncalibrated XGBoost
  picked as winner (lowest Brier on `calib_test`); operating threshold
  `t* = 0.0320` minimizes expected cost under a 30:1 FN:intervention cost ratio.
- **SHAP interpretability** (Phase 8) — `TreeExplainer` on a 5k patient-grouped
  sample, global importance + beeswarm + top-4 dependence plots.
- **Fairness audit** (Phase 9) — Fairlearn parity gaps + per-slice metrics for
  sex / race / age_bin at the deployed threshold.
- **FastAPI service** (Phase 10) — `src/readmit_bench/api/` with `Predictor`
  loader, Pydantic v2 schemas, four endpoints (`/health`, `/model_info`,
  `/predict`, `/predict_batch`). Process-local single-instance load on startup.
- **Docker / HF Spaces** (Phase 10) — `Dockerfile` (Python 3.11-slim, port
  7860) + `requirements-api.txt` + `README_HF.md` HF Spaces metadata header.
- **Test suite** (Phase 11) — 45 pytest tests across all modules; API-tier
  split into 11 always-on schema/contract tests + 5 TestClient integration
  tests gated by a `requires_trained_model` skip marker.
- **CI** (Phase 11) — `.github/workflows/ci.yml` with three jobs:
  `lint` (ruff + black), `test` (matrix Py 3.11 + 3.12), `docker-build`
  (validates Dockerfile builds with stub model artifacts).
- **Documentation** (Phase 12) — `README.md` with full results tables and
  honest-read commentary on every phase; `MODEL_CARD.md` with intended use,
  out-of-scope use, training data caveats, fairness analysis, and ethical
  considerations; this `CHANGELOG.md`.

### Notes & honest reads carried into v1

- All boosted models saturate near **PR-AUC ≈ 0.171** on this cohort. Tuning
  bought essentially zero val lift — the dataset is **capacity-limited, not
  optimization-limited**.
- The deployed calibrator is the **identity** — XGBoost trained with log-loss
  is already well-calibrated on this data; isotonic and Platt slightly worsened
  Brier on `calib_test`.
- The 30:1 cost ratio pushes the optimal policy toward broad screening — at
  `t* = 0.0320` the model flags **87% of encounters** but catches **97% of
  true readmissions**. Operators with a different cost ratio must re-pick the
  threshold from `21_cost_surface.png`.
- Race fairness gap of ~3 pp FNR concentrated on the small "Other" slice
  (n = 3,313); recommended action is monitoring rather than slice-specific
  threshold tuning at low-n.

## [2.0.0] — 2026-04 — V2 SHIPS 🚀

V2 layers four extra capability blocks on top of the v1 production model.
Nothing in v1 changed — the deployed FastAPI service still serves the same
tuned XGBoost + identity calibrator + `t* = 0.0320` threshold. Everything
below is *additive* and lives behind its own driver script.

### Added

- **Neural & AutoML benchmark** (Phase 13) — `src/readmit_bench/nn/`
  - PyTorch tabular MLP, TabNet (`pytorch-tabnet`), FT-Transformer (custom),
    and FLAML AutoML (300 s budget) trained on a 100k patient-grouped
    subsample, scored on the full 198k-row val split.
  - Leaderboard written to `reports/v2_leaderboard.csv` and curves to
    `reports/v2_curves.npz`. MLflow experiment `readmit-bench-v2-nn`.
  - **Honest finding**: NN+AutoML PR-AUC tops out at **0.157** (MLP) versus
    tuned-GBM **0.172** — i.e. tabular GBMs are still the right tool for
    this cohort. Surfaced as a positive negative result in the README.
- **Stacking + voting ensembles** (Phase 14) — `src/readmit_bench/ensembles/`
  - Voting (mean) and stacking (logistic-regression meta-learner) over the
    tuned XGBoost / CatBoost / HistGB triple, fit on a held-out half of val
    using the **same patient-grouped split as Phase 7** (no leakage into the
    meta-learner).
  - **Honest finding**: lift over best base = **+0.00007 PR-AUC**. Base
    learners are too correlated (ρ ≈ 0.99). Stacking LR coefficients land at
    ~3.6 each → essentially equal weighting.
- **LIME + Evidently drift** (Phase 15) — `src/readmit_bench/explainability/`
  + `src/readmit_bench/drift/`
  - LIME local explanations for 3 highest-risk + 3 lowest-risk val encounters
    → `reports/lime/{high,low}_risk_{0,1,2}.html` + `lime_summary.csv`.
  - Evidently 0.7 `DataDriftPreset` train-vs-val report at `reports/drift_report.html`.
- **Flask dashboard** (Phase 16) — `flask_app.py` + `templates/index.html`
  + `static/style.css` + `Dockerfile.flask` + `requirements-flask.txt` + `Procfile`
  - Four tabs: Score (interactive form bound to the production `Predictor`,
    plus JSON `/api/predict` endpoint), Leaderboards, Fairness, Drift (embeds
    the Evidently HTML at `/drift`).
  - Ships as a separate HF Space (Docker SDK) alongside the FastAPI Space, or
    to Render via the included `Procfile` (`gunicorn flask_app:app`).
- **V2 plots** (Phase 17) — `src/readmit_bench/nn/plots.py`
  - Plot 29: V2 NN+AutoML leaderboard with V1 winner reference line.
  - Plot 30: ensembles vs best base, zoomed PR-AUC bar chart.

### Notes & honest reads carried into v2

- **NNs do not beat GBMs here**, full stop. This is a tabular dataset with
  ~100 engineered features and ~10% prevalence — exactly the regime where
  XGBoost dominates. Reported transparently rather than buried.
- **Ensembles do not lift either**. When three base learners agree on 99% of
  ranks the meta-learner has nothing to learn. Stacking LR essentially
  averages the three.
- The drift report is **train vs val sampled** — no synthetic shift was
  injected. It is a baseline artifact for production monitoring, not a
  finding by itself.
